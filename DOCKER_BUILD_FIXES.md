# Docker Build Fixes

## Issues Fixed

### 1. Image Naming (<none> tags)

**Problem**: When running `build.sh`, new Docker images were created with `<none>` tags instead of proper image names, indicating the build process was creating unnamed images each time.

**Root Cause**: The `docker-compose.yml` service definition for `web` did not have an explicit `image:` field. Without this, Docker's default naming scheme would create images without proper tags, leading to orphaned `<none>` images accumulating over time.

**Fix**: Added explicit image naming to `docker-compose.yml`:
```yaml
x-app-base: &app-base
  build: .
  image: rakshagis:web  # ← Added this line
  # ... rest of config
```

Now all builds consistently tag the image as `rakshagis:web`, allowing Docker to reuse and update the same image instead of creating new ones.

### 2. Migrations Permission Error

**Problem**: When running `build.sh`, the following error occurred:
```
PermissionError: [Errno 13] Permission denied: 
'/app/apps/backups/migrations/0002_alter_backupjob_id_alter_backupjob_org_and_more.py'
```

**Root Cause**: The build.sh explicitly called `docker compose run --rm web python manage.py makemigrations --no-input` within a container that had a bind-mounted source directory (`.:/app`). When the container's non-root user (raksha) tried to write to the migrations directory on the host filesystem (owned by a different user), permission was denied. This is particularly problematic on WSL2 and Docker Desktop where file permission mapping is complex.

**Fix**: Implemented a three-part solution:

#### a. Removed explicit makemigrations from build.sh
- Build.sh now skips the `makemigrations` step and relies on the entrypoint.sh to handle it
- Users who add new models should run `python manage.py makemigrations` on the host before deploying

#### b. Updated entrypoint.sh
- The entrypoint now gracefully handles makemigrations failures with better error messages
- If migrations already exist (the common case), it continues without error
- Permission errors are suppressed since migrations should already be in the repo

#### c. Added user guidance
- Build.sh now includes a note explaining the change
- Updated documentation (this file) with instructions

## How to Use After This Fix

### Normal Operation (no new models)
```bash
./build.sh
```
Works as before. No changes needed.

### Adding New Django Models
1. Create your model in the appropriate `apps/*/models.py`
2. Run makemigrations **on the host** (not in Docker):
   ```bash
   python manage.py makemigrations
   ```
3. Run migrations on the host or commit and deploy:
   ```bash
   python manage.py migrate
   ```
4. Then run the build:
   ```bash
   ./build.sh
   ```

### Verifying the Fix

After running `build.sh`, check that images are properly tagged:
```bash
docker image ls | grep rakshagis
```

You should see:
```
rakshagis                    web         <image-id>   ...
```

NOT:
```
<none>                       <none>      <image-id>   ...
```

## Technical Details

### Why This Approach?

- **Explicit image naming** ensures Docker can reuse images instead of creating new ones
- **Moving makemigrations to the host** avoids bind-mount permission issues entirely
- **Graceful entrypoint handling** allows migrations to still be created in production if needed
- **Backward compatible** — existing deployments continue to work

### WSL2 Specifics

On WSL2, file permissions between Windows and Linux are complex. By avoiding container writes to bind-mounted directories during build, we sidestep these issues entirely. Migrations are auto-generated by Django and should be committed to the repository anyway.

### Production Considerations

In production deployments:
1. Migrations should already exist in the Docker image (COPY from Dockerfile)
2. The entrypoint attempts makemigrations as a safety net, but it's non-critical
3. Bind mounts are not used in production, so permissions are not an issue
4. The migrate step still runs and applies any new migrations
